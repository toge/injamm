# Workflow
- Run benchmarks multiple times (3-5 iterations) to verify performance claims, not single runs. Confidence: 0.80
- Compare against baseline numbers and report change rate, not just raw values. Confidence: 0.75
- Present feature additions as a decision with tradeoffs (merits/demerits) before asking for user choice. Confidence: 0.70
- Before `git checkout` / `git restore` for rollback, check `git status` and confirm exactly which files will be reverted; staged-but-unrelated files can be cascaded-reverted unintentionally. Confidence: 0.85
- For complex rollbacks spanning multiple features, confirm with the user before reverting non-target files (especially staged test files that may belong to a different feature branch of work). Confidence: 0.80
- When feature implementation has technical blockers, surface blockers and stop for user decision BEFORE attempting destructive rollback of unrelated in-progress work. Confidence: 0.75
- For multi-feature work, implement and benchmark each feature incrementally (one at a time), not all at once at the end. Confidence: 0.85
- Re-run benchmarks when a single result exceeds the ±10% budget to rule out system noise before drawing conclusions about regression. Confidence: 0.75
- Before `git stash pop`, run `git stash list` to check for pre-existing WIP stashes; conflicting stashes can cause silent work loss when pop fails and `git checkout` is used to "recover". Confidence: 0.85
- When a multi-file refactor is in progress, commit intermediate checkpoints (or save `.bak` copies of touched files) so a failed `git stash pop` doesn't cascade-revert unrelated WIP. Confidence: 0.80
- After `edit_file` reports success on a multi-file bulk rename, verify with `grep` that all expected substitutions landed; on mismatch, fall back to `sed`/`perl -i` for byte-exact replacement. Confidence: 0.75
- For non-UTF-8 source files (e.g., CP932 Japanese), use `perl -CSD -i.bak -pe` with ASCII-only regex substitutions to avoid round-tripping through UTF-8 and corrupting the original encoding. Confidence: 0.75
- For C/C++ performance investigations, use `perf record -g --call-graph dwarf` + `perf report --stdio --percent-limit 1` to identify actual hotspots (e.g., `__memmove_avx512_unaligned_erms`, `_M_replace`) before guessing at optimizations. Re-profile after each change to confirm the targeted hotspot was actually reduced. Confidence: 0.85
- When extending a `bc_var_ref`/struct with a new compile-time-computed field (e.g., `has_dot`), update ALL constructors that build the struct: runtime `add_var_ref`, CT `add_var_ref`, and `to_bytecode` conversion. Missing one path causes test failures in only the CT or runtime variant. Confidence: 0.80
- After bulk-renaming a function signature parameter via `edit_file` + `sed`, manually grep the file for any leftover instances where a same-named local variable (e.g., `ref` vs `var_ref`) was used instead of the correct ref, since sed cannot disambiguate by context. Confidence: 0.75
- When tuning `std::string::reserve` heuristics, profile the actual workload first: a too-small reserve causes `__memmove_avx512_unaligned_erms` on growth; a too-large reserve wastes memory but is nearly free in CPU. Prefer slight over-reservation (e.g., `literal_total_size * 8` with a 256-byte floor) when output is dominated by variable expansions rather than literals. Confidence: 0.75
